# V2.9.5预览版概述

> V2.9.4旨在功能优化，解决历史提交审查中存在的BUG
>
> > 此版本仅为预览版本，为测评本系统性能而存在，虽然此版本在一定程度上修复了历史提交审查存在的问题，但是扫描器扫描，工具依然存在一些严重的逻辑错误，比如，扫描器扫描文件的是当前版本文件，工具调用的文件结果无法正确落在选定范围
> >
> > 因此历史提交的审查能力是不如其他模式的，为了最大发挥本系统的能力，强烈使用其他模式。

此版本相较于V2.9.3版本新增通过PR URL定位到目标提交并进行代码审查，支持一键创建分支并提交PR

在功能界面新增Github PR 审查，此功能开发目的只为复现不同测评数据，此版本存在严重的性能问题和逻辑问题，请只在复现测评数据使用

## 如何使用

在使用GitHub PR 审查前需要选择目标项目，否则无法正常进行

PR链接来源来自GitHub PR，系统将会自动解析并定位到【代码变更】选中两个目标提交

需要注意的是，由于进行深入优化，在系统自动跳转【代码变更】后有小概率出现以下问题：

* 检测到范围正确，但是最终显示的文件数量上千甚至更多
* 进行【代码审查】出现卡死，或者API显示请求体太大，Token限制等问题，通常是上一个问题导致的，系统错误将几千个文件进行解析并发起请求，有时候【代码变更】正确检测范围并展示正确的文件数量也有极小概率出现，概览是叠加的
* 得到审查报告后右上角发起PR部分功能无法正常运行（未全面进行测试）
* 发起PR后，通常来说是一个分支一个PR，分支遵循自动命名，但有时这个功能失效，需手动命名
* PR中需要填写的内容未进行测试，默认填空则提交审查报告（未测试其他情况）
* 在历史会话中提交PR前需要进行一次 PR URL解析，否则有概览失败
* 配置远程分支后，前端可能不会及时更新状态，以实际为准，这不影响提交
* 通过PR获取到的提交范围建议是源仓库的范围，使用Fork的目标仓库有时会出现找不到目标提交范围的情况
  * 比如B是A的Fork，仓库PR来自B仓库，那么应该Git Clone B，如果是Git Clone A 有概率出现未能定位的情况

## 步骤

1. 先fork需要测试的仓库，再git clone
2. 启动本系统进行PR URL解析
3. 进入【代码审查】进行审查
4. （可选）提交PR
